## 1. Executive Summary

**Общая оценка: Сильный MVP с зонами роста**

Watchtower демонстрирует зрелую архитектуру, хорошее качество кода и правильные инженерные решения. Ядро продукта (Slack sync → AI analysis → CLI/Desktop) работает надёжно. Основные риски: безопасность хранения credentials, несогласованность обработки ошибок, и существенный разрыв между стратегическим планом и текущей реализацией (~35-40% от roadmap).

| Область | Оценка | Ключевой риск |
|---------|--------|---------------|
| Архитектура | 8/10 | God-пакет `db/`, circular import risk |
| Go Backend | B+ | Непоследовательный `sql.ErrNoRows` |
| Swift Desktop | Отлично | Подавление ошибок в catch {} |
| Безопасность | MEDIUM | OAuth creds в .env, токены в plaintext |
| Документация | 70% | Chains/Guide не задокументированы |
| Стратегия | 35-40% | Enterprise/multi-LLM/Teams не начаты |

## 2. Архитектура

### 2.1 Что сделано хорошо

- **Чистое разделение CLI / Desktop** через общую SQLite БД (single source of truth)
- **WAL mode** правильно настроен для concurrent reads между CLI и Desktop
- **Pipeline-архитектура** консистентна: digest, tracks, chains, guide — все следуют единому паттерну (Pipeline struct + Run() + Generator interface)
- **Subprocess-based AI** — Claude CLI вызывается как подпроцесс, что упрощает auth и позволяет обоим компонентам работать независимо
- **Worker pool** с приоритетами и context cancellation — корректная concurrent модель

### 2.2 Находки

#### CRITICAL: Риск circular import

`chains/pipeline.go` импортирует `digest`, а `digest/pipeline.go` использует `ChainLinker` interface для обратной связи. Если chains когда-либо понадобятся типы из digest напрямую — цикл сломает компиляцию.

**Рекомендация:** Создать `internal/types/pipeline.go` с shared interfaces (Generator, ChainLinker, ProgressFunc).

#### HIGH: God-пакет `internal/db/`

- 50+ функций на файл, 27 struct-моделей, всё в одном пакете
- Файлы: db.go (800+ строк миграций), models.go, digests.go, tracks.go, channels.go и др.

**Рекомендация:** Разбить на субпакеты: `db/slack/` (users, channels, messages), `db/analysis/` (digests, tracks, chains), `db/people/` (analyses, guides). Или ввести паттерн Store interface (как в prompts.Store).

#### HIGH: Throttle state демона не персистится

`lastPeople` и `lastTracks` в daemon.go — in-memory переменные. При рестарте демона таймеры сбрасываются, что может вызвать повторный запуск daily пайплайнов.

**Рекомендация:** Персистить timestamps в checkpoint таблицу или JSON-файл.

#### MEDIUM: Chains промпт захардкожен

В отличие от digest/tracks/guide, chains pipeline не использует `prompts.Store`. Пользователь не может тюнить chains prompt через `watchtower prompts edit`.

**Рекомендация:** Добавить `chains.linking` и `chains.team_summary` в `prompts.Defaults`.

#### MEDIUM: Несогласованная обработка ошибок в пайплайнах

- Digest pipeline: логирует ошибку, продолжает обработку остальных каналов
- Tracks pipeline: останавливается на первой ошибке
- Guide pipeline: останавливается на первой ошибке

**Рекомендация:** Унифицировать поведение. Добавить `ContinueOnError` config option или `OnError` callback.

#### MEDIUM: Schema migration complexity

800+ строк последовательных миграций в db.go (v1 → v28). Нет rollback, нет тестов на идемпотентность миграций.

**Рекомендация:** Вынести миграции в отдельные файлы. Добавить тесты forward-миграций.

## 3. Go Backend

### 3.1 Что сделано хорошо

- **Error wrapping**: 501 место с `fmt.Errorf("...: %w", err)` — отличная трассировка
- **Goroutine management**: Все 9 горутин в production коде корректно управляют lifecycle (defer wg.Done, context cancellation, channel closing)
- **SQL безопасность**: Все user-derived значения через параметризованные запросы (`?` placeholders)
- **Transaction handling**: Паттерн `defer tx.Rollback()` + explicit Commit() — корректный
- **Rate limiting**: Token bucket + exponential backoff + jitter для Slack API
- **Subprocess management**: Claude CLI с graceful shutdown (SIGINT → 5s → SIGKILL), stderr capped at 64KB

### 3.2 Находки

#### MAJOR: Непоследовательная проверка `sql.ErrNoRows`

Два паттерна используются параллельно:
- `errors.Is(err, sql.ErrNoRows)` — корректный (в workspace.go, profile.go)
- `err == sql.ErrNoRows` — небезопасный при error wrapping (в 13+ файлах)

**Затронутые файлы:**
- `internal/db/tracks.go:165`
- `internal/db/channels.go:124`
- `internal/db/user_analyses.go:130, 1058`
- `internal/db/prompts.go:20, 62, 90, 144`
- `internal/db/sync_state.go:20`
- `internal/db/digests.go:117`

**Рекомендация:** Глобальная замена на `errors.Is()` во всех файлах.

#### MINOR: Микс log.Logger и slog.Logger

Большинство кода использует `log.Logger`, но `prompts/store.go` использует `slog.Logger`. Не критично, но непоследовательно.

#### MINOR: Динамический SQL в двух местах

- `messages.go:97` — `msgColumns()` + `fromTable` (безопасно, hardcoded values)
- `db.go:785` — `fmt.Sprintf` для column names в миграции (безопасно, hardcoded slice)

Реальной уязвимости нет, но паттерн нетипичен для idiomatic Go.

## 4. Swift Desktop

### 4.1 Что сделано хорошо

- **MVVM архитектура**: Чистое разделение Models → Queries → ViewModels (@MainActor, @Observable) → Views
- **GRDB integration**: DatabasePool с WAL, Sendable conformance, thread-safe queries
- **Memory management**: Корректное использование `[weak self]` в Task closures, cancellation patterns
- **Subprocess security**: JSONSerialization для MCP config (предотвращает injection)
- **Onboarding**: State machine с persistence через UserDefaults, хорошая UX

### 4.2 Находки

#### CRITICAL: Массовое подавление ошибок

Множество `catch {}` блоков без логирования:

- `ChatHistoryViewModel.swift`: строки 46, 60, 73, 83, 94
- `ChatViewModel.swift`: строка 202 — комментарий "silently ignore"
- `DashboardViewModel.swift`: строка 45

**Влияние:** Крайне затрудняет debugging. Ошибки исчезают бесследно.

**Рекомендация:** Заменить все `catch {}` на `catch { NSLog("[Component] error: \(error)") }`.

#### MAJOR: Shell subprocess при запуске

`Constants.swift:114-150` — `resolvedEnvironment()` запускает shell subprocess для получения PATH. Таймаут 5 секунд, но выполняется синхронно. Если shell завис — app замёрзнет.

**Рекомендация:** Перенести на background thread с lazy initialization.

#### MAJOR: Regex на каждый frame

`MessageTextView.swift:27` — `NSRegularExpression` компилируется при каждом вызове body view.

**Рекомендация:** Вынести в `static let` property.

#### MINOR: Static shared AppState для уведомлений

`WatchtowerApp.swift:11` — `NotificationDelegate.sharedAppState` устанавливается в `.onAppear`. Если уведомление приходит до `.onAppear`, reference будет nil.

#### MINOR: Отсутствие VoiceOver labels

Icon-only кнопки (TracksListView, DigestListView и др.) не имеют accessibility labels.

#### MINOR: Нет таймаута на Claude CLI subprocess

В отличие от DB migration (30-секундный таймаут), Claude stream не имеет таймаута. Может зависнуть бесконечно.

## 5. Безопасность

### 5.1 Что сделано хорошо

- **SQL injection**: Все запросы параметризованы, FTS5 sanitized через `sanitizeFTS5Query()`
- **Command injection**: `exec.CommandContext` с массивом аргументов (не shell string)
- **File permissions**: Config 0o600, logs 0o600, PID 0o600, DB directory 0o700
- **Path traversal**: Workspace name валидируется regex `^[a-zA-Z0-9][a-zA-Z0-9_.-]*$`
- **OAuth CSRF**: State parameter корректно валидируется
- **TLS**: OAuth callback использует TLS 1.2+, Slack API через HTTPS
- **Process security**: PID reuse detection, stale PID cleanup, exclusive file locking

### 5.2 Находки

#### CRITICAL: OAuth credentials в репозитории

Файл `.env` содержит plaintext credentials:
- `WATCHTOWER_OAUTH_CLIENT_SECRET=f50df2...`
- `CODESIGN_IDENTITY="Developer ID Application: ..."`

`.gitignore` исключает `.env*`, но при публикации репо credentials будут скомпрометированы.

**Рекомендация:** Немедленно ротировать OAuth client secret в настройках Slack App. Использовать исключительно environment variables в CI/CD.

#### HIGH: Slack токен в plaintext config

`~/.config/watchtower/config.yaml` хранит `xoxp-*` / `xoxb-*` токены без шифрования. Файл имеет 0o600 permissions (хорошо), но при компрометации home directory — полный доступ к Slack workspace.

**Рекомендация:** Внедрить macOS Keychain для хранения токенов. Оставить config.yaml только для non-sensitive настроек.

#### HIGH: SQLite БД не зашифрована

База данных содержит все сообщения Slack workspace в plaintext. При физическом доступе к устройству — полная утечка корпоративной переписки.

**Рекомендация:** Интегрировать SQLCipher или SQLite encryption extension. Key derivation из Keychain.

#### HIGH: Self-signed TLS cert для OAuth

OAuth callback server использует self-signed certificate. Пользователь должен вручную доверять сертификату (`watchtower auth trust-cert`). Потенциальный MITM при игнорировании SSL warning.

**Рекомендация:** Использовать ASWebAuthenticationSession на macOS для нативного OAuth flow.

#### MEDIUM: Sensitive data в логах

Логи демона (`~/.local/share/watchtower/{workspace}/watchtower.log`) могут содержать контент Slack сообщений при ошибках API. Permissions 0o600 (хорошо), но содержимое не фильтруется.

**Рекомендация:** Добавить structured logging с redaction для message content.

#### LOW: Lock file permissions

`cmd/sync.go:209` — lock file создаётся с 0o644 (world-readable). Должно быть 0o600.

#### LOW: Parent directories с 0o755

`daemon/pidfile.go:22` — `os.MkdirAll(..., 0o755)`. Другие пользователи системы могут читать listing директории.

- **CLI flag conventions** последовательны: `--workspace`, `--config`, `--verbose`

### 6.2 Находки

#### CRITICAL: README устарел

- Строка 44: `watchtower actions` → должно быть `watchtower tracks`
- Строка 196: "Actions" tab → "Tracks"

#### HIGH: Chains фича не задокументирована

Реализовано: `internal/chains/`, `cmd/chains.go`, ChainDetailView, ChainsListView. В документации — только техническое упоминание в llm-pipelines.md. Нет:
- Что такое "chain"?
- Как chains создаются?
- CLI workflow (`watchtower chains generate/archive`)
- User guide

#### HIGH: Guide фича не задокументирована

Реализовано: `internal/guide/`, `cmd/guide.go`, GuideViewModel, GuideView/. Документация `docs/guide-analysis.md` практически пуста. Нет объяснения что это, зачем, как использовать.

#### HIGH: Нет user guide для Desktop app

`docs/desktop-app.md` — отличная техническая спецификация, но не пользовательский гайд. Нет:
- Quick start для desktop
- Описания навигации и вкладок
- Туториалов по фичам

#### MEDIUM: "Decisions" vs "Chains" — неясная терминология

`watchtower decisions` и `watchtower chains` — два разных CLI-команды. Связь между ними нигде не объяснена. Пользователь не понимает когда использовать какую.

#### MEDIUM: Недокументированные config keys

`digest.language`, `digest.min_messages` — используются в коде, но не описаны в config reference в usage.md.

#### MEDIUM: Нет troubleshooting.md

Типичные проблемы (database locked, claude not found, daemon issues) разбросаны по разным docs. Нет единого руководства.

### 7.3 Стратегический разрыв

Продукт позиционируется как **"multi-source organizational intelligence platform"**, но по факту является **"single-workspace Slack analytics tool"**.

**Оценка отставания:** 8-12 недель от roadmap. Enterprise server — ключевой компонент монетизации — не начат.

**Стратегическое решение необходимо:**
- **Вариант A**: Полная реализация стратегии → 3-4 инженера на 6 месяцев, привлечение капитала
- **Вариант B**: Фокус на freemium/pro для малых команд ($10/user/mo) → текущая команда, меньший TAM
- **Рекомендация**: Вариант A с инкрементальной валидацией. Начать с LLM abstraction layer + enterprise server MVP.
