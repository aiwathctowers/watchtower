# Channel Memory — Running Context для пайплайнов

> Статус: Draft
> Дата: 2026-03-21

## Проблема

Каждый запуск дайджест-пайплайна анализирует сообщения канала **с нуля**, без знания предыдущих анализов. Это приводит к:

1. **Потере контекста** — AI не знает, что тема X обсуждалась вчера, решение Y было принято на прошлой неделе. Каждый дайджест — изолированный снимок.
2. **Повторяющимся выводам** — одно и то же решение попадает в дайджесты день за днём, пока тема активна.
3. **Невозможности отслеживать эволюцию** — "тема X, начатая 3 дня назад, до сих пор не решена" — невыразимо в текущей модели.
4. **Избыточной передаче токенов** — при отсутствии контекста AI вынужден каждый раз строить понимание канала заново из сырых сообщений.

### Текущее состояние

| Пайплайн | Предыдущий контекст | Поведение |
|----------|---------------------|-----------|
| Channel digest | Нет | Каждый анализ изолирован |
| Daily rollup | Нет (видит только channel digests текущего дня) | Не знает про вчера |
| Weekly trends | Нет (видит только daily digests текущей недели) | Не знает про прошлую неделю |
| Tracks | Частично (видит existing tracks + recent decisions) | Дедупликация есть, но без исторического контекста |
| People cards | **Да** (загружает предыдущую карточку) | Единственный пайплайн с "памятью" |

People cards уже реализуют паттерн "передать предыдущий результат в промпт" — нужно распространить его на остальные пайплайны.

## Цели

### Бизнес-цели

1. **Дайджесты показывают динамику, а не статику** — пользователь видит "что изменилось", а не просто "что произошло".
2. **Устранение дублирования** — решение/тема упоминается один раз при появлении, далее — только при изменении статуса.
3. **Более точные tracks** — AI знает историю трека и может обновлять статус, а не создавать дубли.
4. **Снижение стоимости AI** — сжатый контекст (~300-500 токенов) заменяет повторную передачу сырых сообщений предыдущих периодов.

### Метрики успеха

- Снижение дублирования решений в daily/weekly дайджестах на ≥50%
- Появление формулировок типа "continues from...", "resolved since..." в ≥30% дайджестов активных каналов
- Снижение input-токенов на 10-15% для повторных анализов одного канала (меньше нужды дублировать контекст)

## Решение: Channel Memory

### Концепция

После каждого анализа AI генерирует **running summary** — сжатый контекст канала, который передаётся в следующий анализ того же канала. Это управляемая "память", хранимая в БД, а не в сессиях CLI.

```
Цикл:
  1. Загрузить running summary предыдущего периода
  2. Передать в промпт: [running summary] + [новые сообщения]
  3. AI анализирует с учётом контекста
  4. AI генерирует обновлённый running summary
  5. Сохранить summary в БД → goto 1
```

### Что хранится в running summary

Running summary — это структурированный JSON, генерируемый AI как часть ответа на дайджест-промпт.

```json
{
  "active_topics": [
    {
      "topic": "Миграция на PostgreSQL",
      "status": "in_progress",
      "started": "2026-03-18",
      "last_update": "2026-03-21",
      "key_participants": ["U123", "U456"],
      "summary": "Решили мигрировать, Алексей делает POC, блокер — совместимость с legacy API"
    }
  ],
  "recent_decisions": [
    {
      "decision": "Используем pgx вместо database/sql",
      "date": "2026-03-20",
      "by": "U123",
      "status": "active"
    }
  ],
  "channel_dynamics": "Канал активен, основные участники: @alice (техлид, принимает решения), @bob (исполнитель). Тон конструктивный, решения принимаются быстро.",
  "open_questions": [
    "Нужен ли отдельный сервис для миграции или встраиваем в существующий?"
  ],
  "meta": {
    "generated_at": "2026-03-21T10:00:00Z",
    "digest_id": "abc-123",
    "message_count": 47,
    "period": "2026-03-21"
  }
}
```

### Бюджет токенов

| Компонент | Токены | Примечание |
|-----------|--------|------------|
| Running summary (input) | 300-800 | Сжатый контекст предыдущего периода |
| Новые сообщения | 3000-15000 | Как сейчас |
| Running summary (output) | 200-500 | Часть JSON-ответа дайджеста |

Итого: overhead ~500-1300 токенов на вызов. При сокращении дублирования и лучшем качестве — положительный ROI.

## Уровни применения

### Уровень 1: Channel Digests (MVP)

**Что меняется:**
- Перед генерацией channel digest — загрузить `running_summary` предыдущего дайджеста этого канала
- Добавить секцию `=== PREVIOUS CONTEXT ===` в промпт
- AI генерирует обновлённый `running_summary` как часть JSON-ответа
- Сохранить в поле `running_summary` таблицы `digests`

**Промпт-инструкции для AI:**
```
=== PREVIOUS CONTEXT ===
{running_summary JSON}

Rules:
- Use PREVIOUS CONTEXT to detect continuity: evolving topics, resolved questions, changed decisions.
- Say "continues from [date]" for ongoing topics, "resolved since [date]" for closed ones.
- Do NOT repeat decisions/topics from PREVIOUS CONTEXT unless their status changed.
- Generate an updated running_summary reflecting the current state after this analysis.
```

**Влияние на пользователя:**
- Дайджесты становятся инкрементальными: "Вчера начали обсуждать миграцию → сегодня решили использовать pgx → завтра Алексей показал POC"
- Меньше шума от повторяющихся тем

### Уровень 2: Daily/Weekly Rollups

**Что меняется:**
- Daily rollup получает `running_summary` предыдущего daily rollup
- Weekly — предыдущего weekly
- AI может сказать: "На этой неделе закрыли 3 из 5 вопросов прошлой недели"

**Влияние на пользователя:**
- Weekly дайджест показывает прогресс, а не просто список тем
- Тренды становятся реальными трендами (vs повторяющиеся снимки)

### Уровень 3: Tracks Enrichment

**Что меняется:**
- Tracks pipeline получает `running_summary` канала для обогащения контекста
- AI лучше понимает: этот трек — продолжение вчерашнего обсуждения или новая тема

**Влияние на пользователя:**
- Меньше дублирующихся треков
- Более точное определение приоритета (AI знает, как долго тема висит)

## Требования к реализации

### Данные

1. **Новое поле** `running_summary TEXT` в таблице `digests`
   - Хранит JSON running summary, сгенерированный AI
   - NULL для дайджестов, созданных до фичи (graceful degradation)

2. **Новый запрос** `GetLatestRunningContext(channelID, digestType) → string`
   - Возвращает `running_summary` последнего дайджеста для канала/типа
   - Аналог существующего `GetLatestDigest`, но возвращает только summary

### Промпты

3. **Модификация промпта `digest.channel`**
   - Добавить секцию `=== PREVIOUS CONTEXT ===` (опциональную — если summary есть)
   - Добавить инструкции по использованию контекста (см. выше)
   - Добавить `running_summary` в требуемый JSON-output

4. **Модификация промптов `digest.daily` и `digest.weekly`** (уровень 2)
   - Аналогичные изменения

### Пайплайн

5. **Модификация `generateChannelDigest()`**
   - Перед вызовом AI: загрузить предыдущий running summary
   - После вызова AI: сохранить новый running summary вместе с дайджестом

6. **Graceful degradation**
   - Если предыдущего summary нет (первый запуск, новый канал) — промпт работает без секции PREVIOUS CONTEXT
   - Если summary не парсится / повреждён — игнорировать, логировать warning

### Ограничения и защита

7. **Размер running summary** — ограничить ~2000 символов. AI должен сжимать, а не накапливать бесконечно.

8. **Протухание** — если последний дайджест старше 7 дней, running summary передаётся с пометкой `(outdated, from {date})`. Старше 30 дней — не передаётся.

9. **Сброс** — CLI команда `watchtower digest reset-context [channel]` для ручного сброса running summary (если AI зациклился на устаревшем контексте).

## Этапы реализации

### Этап 1: MVP — Channel Digest Memory
- Миграция БД: поле `running_summary` в `digests`
- Запрос `GetLatestRunningContext`
- Модификация промпта `digest.channel`
- Модификация `generateChannelDigest()` в pipeline.go
- Тесты: pipeline с mock-генератором

### Этап 2: Rollup Memory
- Модификация промптов `digest.daily`, `digest.weekly`
- Модификация `generateDailyDigest()`, `generateWeeklyDigest()`
- Тесты

### Этап 3: Tracks Integration
- Передача channel running summary в tracks pipeline
- Модификация промпта `tracks.extract`
- Тесты

### Этап 4: Desktop UI
- Показывать "ongoing topics" из running summary в канальном дайджесте
- Индикатор "continues from..." на дайджестах

## Риски

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| AI генерирует раздутый running summary | Средняя | Лимит 2000 символов, инструкция "compress aggressively" |
| Дрейф контекста (AI зацикливается на теме) | Низкая | TTL 7/30 дней, команда reset-context |
| Ухудшение качества при плохом summary | Низкая | Graceful degradation — без summary работает как раньше |
| Увеличение стоимости AI | Низкая | Overhead ~500-1300 токенов vs потенциальная экономия на качестве |

## Не входит в scope

- Персистентные сессии Claude CLI (подход отвергнут в пользу DB-stored context)
- Per-channel сессии для interactive chat (отдельная фича, если потребуется)
- Автоматическая ротация/архивация running summaries (пока хватит TTL)
- Визуализация эволюции тем (timeline view) — отдельная фича
